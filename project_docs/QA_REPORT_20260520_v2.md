## CRITICAL / HIGH（6 个，影响数据正确性或会 crash）

### B-001 `/statistics/by-grade` 对真实数据永远返回空数组 [HIGH]
- **位置**：`backend/app/api/v1/endpoints/statistics.py:307`
- **原因**：`grade_order = ["A", "B", "C", "D", "F", "N/A"]`，但 DB 中实际 grade 全部带 `?` 后缀（"C+?" / "C?" / "C-?" / 空）
- **数据证据**：
  ```
  C+?: 219 笔
  C?:  206 笔
  C-?: 47 笔
  (空): 53 笔
  ```
- **后果**：Statistics 页"按评级分析"区块**永远空白**。前端没有 empty-state 提示，用户看到的是空 table。
- **建议修复**：grade_order 加上 `["C+?", "C?", "C-?", "?", ""]` 或者剥离 `?` 后做 fuzzy 聚合

### B-002 score_grade 100% 带 `?` 后缀（评分系统失效）[HIGH]
- **位置**：`src/analyzers/quality_scorer.py:1620-1623`
- **设计意图**：`?` 代表"输入不足，评分基本是默认值"
- **实际**：472 个 closed positions 中 **0 个干净 grade**（219+206+47+53 = 525 全带 `?` 或空）
- **后果**：评分系统对真实数据**几乎无效**。要么 input completeness 阈值过严，要么真的所有数据都不全——前者是 bug，后者也得在 UI 上诚实说明
- **建议**：
  1. 调查 `_check_input_completeness()` 阈值
  2. 即使阈值合理，UI 上要明确告诉用户"评分系统当前对你的数据集只能给出降级评分（带 `?`），原因是 X"

### B-003 sharpe_ratio 在两个 endpoint 不一致 [HIGH]
- `/statistics/performance.sharpe_ratio` = **null**
- `/statistics/risk-metrics.sharpe_ratio` = **0.18**
- 同一份数据、同一个指标、两个值。说明两处实现独立且至少一处有 bug
- **后果**：用户看 Statistics 页和 Risk 区块看到的 sharpe 不一样，丧失数据信任
- **建议**：抽出 `compute_sharpe()` 单一来源，两 endpoint 都调用

### B-004 `Backtest.tsx:115` 潜在 `.toFixed()` 崩溃 [HIGH]
- **位置**：`frontend/src/pages/Backtest.tsx:115`
- **代码**：`r.savings_pct.toFixed(0)%`（line 113 有 `!== null` 守卫，但 line 115 没用同一守卫）
- **风险**：极端数据下 race condition 或 schema 变化时会 crash
- **建议**：改成 `{r.savings_pct?.toFixed(0) ?? '-'}%`

### B-005 ai_coach 硬编码过期模型 [HIGH]
- **位置**：`backend/app/services/ai_coach.py:196`
- **代码**：`"model": "claude-3-haiku-20240307"`
- 今天 2026-05-20，这个模型 ID 早已下线/不推荐
- **建议**：改用 `claude-haiku-4-5-20251001`（或 sonnet-4-6 / opus-4-7，根据成本权衡）

### B-006 `/dashboard/needs-review` 后端有、前端没调 [HIGH]
- 后端 `dashboard.py:198` 注册了 `GET /needs-review`
- 前端 `Dashboard.tsx` 没拉取这个 endpoint
- **后果**：需要 review 的持仓在 UI 上看不到——一个完整的产品功能 missing

## MEDIUM（8 个，影响 UX 但不 crash）

### B-007 max_drawdown_pct 始终 null
- `/statistics/performance.max_drawdown_pct` = null
- `/statistics/risk-metrics.max_drawdown_pct` = null
- `max_drawdown = $11625.71` 有值，但 % 形式缺失
- **建议**：用 `max_drawdown / peak_equity` 计算并返回

### B-008 drawdowns 百分比超过 100% 误导
- drawdown #1: `drawdown_pct = 113.89%` (peak=$10208, trough=-$1417)
- drawdown #2: `drawdown_pct = 240.06%`
- 数学上 trough 为负数 → pct 超 100%
- **建议**：分母规范化为 `max(peak, peak + abs(min(0, trough)))` 限上限 100%；或 UI 上加说明"含本金亏损"

### B-009 fees_pct_of_pnl = 309.73% 反人类
- `/statistics/performance.fees_pct_of_pnl = 309.73%`
- 因为净盈亏接近 0（$1035）而手续费 $3207，比值刺眼
- **建议**：改用 `fees_pct_of_gross_pnl`：$3207 / $4242 = **75.6%**（更直观：手续费吃掉总盈亏的 76%）

### B-010 货币混算风险（无 currency 标签）
- DB 有 USD + HKD 混合，但 `/statistics/by-direction`, `by-symbol`, `by-grade`, `by-holding-period` 聚合时不区分 currency
- Frontend `Statistics.tsx:584/703/739/769` 用 `formatCurrency(total_pnl)` 默认按 USD 渲染
- **后果**：港股 HKD 交易被当成 USD 显示
- **建议**：后端在 aggregate 前先统一转 USD（已有 `currency.py::get_pnl_in_usd`），或在 response 加 `currency_breakdown` 字段

### B-011 多页 table 空数据态无提示
- `Statistics.tsx:727` byGrade.map()、`Statistics.tsx:796` drawdowns.slice(0,5).map()
- 后端返回空数组时，table header 还在但 body 空——用户以为加载完了但什么都没
- **建议**：每个 table block 加 `{data.length === 0 && <EmptyState />}`

### B-012 a11y: 5+ 个 onClick `<tr>` 没 role/tabIndex
- `Positions.tsx:391`, `Statistics.tsx:698/727/764/796`
- 键盘用户无法 drill-down
- **建议**：统一加 `role="button" tabIndex={0} onKeyDown={...}`

### B-013 Dashboard 部分元素无 dark: 变体
- `Dashboard.tsx:55-58` heading 用 `text-gray-900` / `text-gray-500` 没配 dark:
- 暗模式下颜色对比不足
- **建议**：补 `dark:text-white` / `dark:text-gray-400`

### B-014 Backtest savings_pct 误导
- `cf5_hard_stop_10pct.savings_pct = 1477.88%`（$15302 / $1035）
- 因为分母（实际盈亏）太小导致 % 失真
- **后果**：用户看到"+1477%"以为有 huge 提升，实际是 $15K 增益但 base 接近 0
- **建议**：UI 上**优先展示绝对金额（$15,302）**，% 作为次要；或者用 `savings / abs(skipped_pnl)` 作为分母
